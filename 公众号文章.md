# 利用 GitHub Actions 自动保持 GitHub 提交状态常绿 —— 详细操作教程

> **阅读本文，你将学会：** 无需服务器、无需写代码，利用 GitHub Actions 免费额度，让你的 GitHub 提交记录面板保持常绿。全程 10 分钟即可完成部署。

---

## 一、为什么你的 GitHub 贡献面板是"荒漠"？

打开 GitHub 个人主页，那个绿白相间的 Contribution Graph（贡献热力图）是技术面试官最常瞄一眼的地方。

但现实是：

- 工作用的仓库在私有 GitLab / Gitee 上，GitHub 收不到提交
- 忙起来一周不写代码，面板直接"白茫茫一片"
- 周末和假期更是一片空白

**这不代表你不 coding**，但在不了解你的人眼里，那块面板就是你的"技术活跃度"。

于是就有了 **Github Auto Commit** —— 一个开源工具，帮你自动、持续地向 GitHub 仓库生成真实感的提交记录。

> 🔗 项目地址：**[https://github.com/hackliu/Github_Auto_Commit](https://github.com/hackliu/Github_Auto_Commit)**

## 二、这个工具有什么特别？

市面上的"GitHub 自动提交"工具不少，但它们大多：

- 配置硬编码在脚本里，改个频率还得翻代码
- Token 明文写在文件中，一不留神就提交到公开仓库
- 提交内容永远是同一条 "update"，机器人痕迹明显
- 只支持 crontab，换 Windows 就没法用

**Github Auto Commit 的解决方式：**

```
┌─────────────────────────────────────────────────────────┐
│                    Github Auto Commit                    │
├─────────────────────────────────────────────────────────┤
│  🔒 安全      Token 仅存 .env，日志中自动脱敏为 ***      │
│  🎲 真实感    24种日志短语 + 4种Markdown模板 + 20种Emoji │
│  ⚙️ 可配置    YAML 驱动，改参数无需读代码                │
│  🖥️ 跨平台    Windows / macOS / Linux / Docker / Actions │
│  🛡️ 容错      指数退避重试 + Push冲突自动rebase          │
│  🚫 防误伤    远程仓库URL校验，绝不碰其他项目            │
└─────────────────────────────────────────────────────────┘
```

**核心工作流程：**

```
  GitHub 定时器触发（每30分钟）
        │
        ▼
  Actions Runner 启动
        │
        ├─ 安装 Python 依赖
        ├─ 读取 config.yaml
        ├─ 随机选择一个文件（如 activity.log）
        ├─ 随机选择一条提交消息
        ├─ 生成随机内容（日志行 / Markdown 页脚 / Emoji）
        ├─ git commit → git push
        └─ Runner 销毁
```

最关键的一点：**完全免费**。GitHub 个人账户每月有 2,000 分钟 Actions 额度，本工具每次运行只需 6 秒，就算每天跑 48 次，一个月也只用不到 750 分钟。

## 三、10 分钟部署教程

### 准备工作清单

在开始之前，确保你具备：

| 序号 | 准备项 | 说明 |
|:---:|--------|------|
| 1 | 一个 GitHub 账号 | 没有的话去 [github.com](https://github.com) 免费注册 |
| 2 | 一个目标仓库 | 即你希望有提交记录的那个仓库（可以是新建的空仓库） |
| 3 | 10 分钟时间 | 跟着本文操作即可 |

### 第一步：创建目标仓库（如果你还没有）

1. 打开 GitHub，点击右上角 **+** → **New repository**
2. Repository name：`my-project`（或任意你喜欢的名字）
3. 选择 **Public**（公开仓库贡献面板才显示）
4. 勾选 **Add a README file**
5. 点击 **Create repository**

```
┌──────────────────────────────────────┐
│  Create a new repository             │
│                                      │
│  Owner:  hackliu                     │
│  Repo:   my-project                  │
│  ● Public    ○ Private               │
│  ☑ Add a README file                 │
│                                      │
│      [ Create repository ]           │
└──────────────────────────────────────┘
```

### 第二步：获取项目代码

打开 **[https://github.com/hackliu/Github_Auto_Commit](https://github.com/hackliu/Github_Auto_Commit)**，点击绿色 **Code** 按钮 → **Download ZIP**，解压到任意目录。

或者使用 Git 命令：

```bash
git clone https://github.com/hackliu/Github_Auto_Commit.git
cd Github_Auto_Commit
```

### 第三步：生成 GitHub Token

> ⚠️ **这是最关键的安全步骤**——Token 相当于你的 GitHub 密码，**切勿**泄露给任何人，也**不要**直接写在配置文件里。

1. 登录 GitHub → 右上角头像 → **Settings**
2. 左侧菜单最底部 → **Developer settings** → **Personal access tokens** → **Fine-grained tokens**
3. 点击 **Generate new token**

按以下配置填写：

```
┌────────────────────────────────────────────────┐
│  Token name:  Github Auto Commit               │
│  Expiration:  90 days (建议定期更换)             │
│  Resource owner:  hackliu (你的用户名)           │
│                                                │
│  Repository access:                            │
│    ● Only select repositories                  │
│    选择: my-project (你的目标仓库)               │
│                                                │
│  Permissions:                                  │
│    Contents:  Read and Write  ← 必须！          │
│                                                │
│      [ Generate token ]                        │
└────────────────────────────────────────────────┘
```

4. 生成后**立即复制** Token（格式：`github_pat_11AB...`），它只显示这一次！

### 第四步：配置项目文件

进入解压后的 `Github_Auto_Commit` 目录，修改两个文件：

#### 4.1 创建 .env 文件

```bash
# 复制模板
cp .env.example .env
```

用记事本（或任意编辑器）打开 `.env`，替换为你的 Token：

```ini
GIT_AUTO_COMMIT_TOKEN=github_pat_你的Token值
```

#### 4.2 修改 config.yaml

打开 `config.yaml`，重点修改以下几行：

```yaml
git:
  # ← 改成你的目标仓库地址
  repo_url: "https://github.com/hackliu/my-project.git"
  branch: "main"

commit:
  min_daily: 1          # 每天最少提交 1 次
  max_daily: 5          # 每天最多提交 5 次（实际在 1~5 之间随机）
  min_interval_minutes: 60
  max_interval_minutes: 240
  files:
    - "activity.log"    # 会被自动修改的文件
    - "README.md"       # 会被自动修改的文件

active_hours: [8, 23]   # 北京时间 8:00~22:59 之间活动
skip_weekends: true      # 周六日休息，更像真人
```

#### 4.3 在目标仓库中创建 activity.log

在你的**目标仓库**（my-project）中，创建一个空的 `activity.log` 文件：

```bash
# 克隆目标仓库
git clone https://github.com/hackliu/my-project.git
cd my-project

# 创建空文件
echo "# Activity Log" > activity.log

# 提交
git add activity.log
git commit -m "Add activity log"
git push
```

### 第五步：将项目文件放入目标仓库

将 `Github_Auto_Commit` 目录中的以下文件/文件夹，复制到你的目标仓库：

```
需要复制的文件：
├── src/                  ← 整个文件夹
├── .github/workflows/    ← 整个文件夹（含 schedule.yml）
├── requirements.txt
├── config.yaml
└── .gitignore
```

> ❌ **不要复制** `.env` 文件！它已在 `.gitignore` 中排除，Token 不应该出现在仓库中。

复制后的目标仓库目录结构：

```
my-project/
├── src/                    ← 自动化脚本
├── .github/workflows/
│   └── schedule.yml        ← 定时触发器
├── config.yaml             ← 你的配置
├── requirements.txt
├── .gitignore
├── activity.log            ← 会被自动修改
├── README.md               ← 你的 README
└── ... (其他项目文件)
```

提交并推送：

```bash
git add .
git commit -m "Add Github Auto Commit workflow"
git push origin main
```

### 第六步：配置 GitHub Secret

这是**安全关键步骤**——Token 通过 GitHub 的加密 Secret 机制存储，运行时会自动隐藏。

1. 打开目标仓库页面 → 顶部 **Settings** 标签
2. 左侧菜单 → **Secrets and variables** → **Actions**
3. 点击绿色按钮 **New repository secret**
4. 填写：

```
Name:   GIT_AUTO_COMMIT_TOKEN
Secret: github_pat_11AB...（第四步复制的 Token）
```

> 📸 界面示意：
> ```
> ┌─────────────────────────────────────────┐
> │  Actions secrets and variables          │
> │                                         │
> │  Secrets  │  Variables                  │
> │                                         │
> │  ┌─────────────────────────────────┐    │
> │  │ GIT_AUTO_COMMIT_TOKEN           │    │
> │  │ ******************************  │    │
> │  │              [ Update ]         │    │
> │  └─────────────────────────────────┘    │
> │                                         │
> │    [ New repository secret ]            │
> └─────────────────────────────────────────┘
> ```

1. 打开仓库 → **Actions** 标签
2. 左侧找到 **"Git Auto Commit"**
3. 点击右侧 **"Run workflow"** → 选择 `main` 分支 → **"Run workflow"**（手动触发）

## 五、进阶配置

### 5.1 让提交看起来更真实

```yaml
# config.yaml 进阶配置
commit:
  min_daily: 0          # 允许某天没有提交（真实开发者也会休息）
  max_daily: 6          # 最多 6 次
  min_interval_minutes: 45
  max_interval_minutes: 300  # 间隔可达 5 小时
  messages:              # 自定义更专业的提交消息
    - "fix: resolve edge case in data parser"
    - "feat: add pagination to API response"
    - "refactor: extract common validation logic"
    - "docs: update API documentation"
    - "test: add unit tests for user module"
    - "chore: update dependencies"
  files:
    - "activity.log"
    - "CHANGELOG.md"     # 多个文件轮流修改

active_hours: [9, 21]   # 早9点到晚9点
skip_weekends: true      # 周末不提交
```

### 5.2 调整触发频率

编辑 `.github/workflows/schedule.yml` 中的 cron：

```yaml
on:
  schedule:
    # 修改这里：
    # ┌─分钟 ┬─小时 ┬─日 ┬─月 ┬─星期
    - cron: "7 * * * *"    # 每小时 1 次（低调模式）
    # - cron: "7,37 * * * *"  # 每小时 2 次（标准模式）
```

> ⚠️ GitHub Actions 使用 **UTC 时间**。北京时间 = UTC + 8。例如 `active_hours: [9, 23]` 在北京时间是 9:00~22:59，对应 UTC 的 1:00~14:59。

### 5.3 多个仓库同时挂机

```bash
# 为每个目标仓库准备独立的配置文件
python -m src.main --daemon --config config-repo-a.yaml &
python -m src.main --daemon --config config-repo-b.yaml &
```

## 六、常见问题速查

**Q: 会不会被封号？**

A: 本工具设计为适度使用（每天 0~8 次提交）。这个频率和正常开发者完全一致，GitHub 不会对个人开发者做这种频率的检查。但如果你设置成每分钟提交一次，那就不一样了。

**Q: 为什么 Actions 没有触发？**

A: 检查三点：① `.github/workflows/schedule.yml` 文件路径是否正确（必须在仓库的 `.github/workflows/` 目录下）；② 文件名是否以 `.yml` 结尾（不是 `.yaml`）；③ 先手动点 "Run workflow" 测试。

**Q: 日志显示 "403 Forbidden"？**

A: Token 权限不足。检查你的 Fine-grained Token 是否有目标仓库的 `Contents: Read and Write` 权限。

**Q: 仓库已有提交，但贡献面板没绿？**

A: 贡献面板只统计以下提交：① 提交到**公开**仓库的默认分支或 `gh-pages` 分支；② 提交者的邮箱与 GitHub 账户关联。确认这些条件满足。

**Q: 在中国大陆能正常使用吗？**

A: GitHub Actions 的 Runner 在境外，**天然不受网络限制**。如果你是在本地运行（非 Actions 方式），可参考项目 README 的 §8 章节解决网络问题。
